# Job Operational ID Backend Scope

## Goal

Add a friendly, public-facing job operational ID that is separate from the internal job UUID.

The internal UUID remains the primary key and system identity. The operational ID is used for user-facing references, support conversations, exports, emails, and job search.

## Decision Summary

Use a system-generated operational ID with this format:

```text
YYMMDDSSSSS
```

Example:

```text
26072100001
```

Where:

```text
YY    = two-digit creation year
MM    = two-digit creation month
DD    = two-digit creation day
SSSSS = zero-padded global daily sequence
```

Do not expose the job UUID as the operational/job reference in user-facing areas.

## Key Rules

- The UUID remains the internal primary key.
- The operational ID is generated by the backend only.
- Users and organization admins do not set, edit, or reset operational IDs.
- No text prefix such as `J-` or `JOB-`.
- No organization/company code in the operational ID.
- No hour component.
- The sequence is global per day across the platform.
- The operational ID is assigned once at job creation.
- The operational ID does not change if origin, fulfillment, or ownership fields change later.
- Existing jobs must receive generated operational IDs during migration.

## Why This Format

This codebase does not have a `companies` table or a single `jobs.company_id`. Jobs use three organization axes:

```text
jobs.origin_org_id
jobs.customer_owner_org_id
jobs.fulfillment_org_id
```

Those fields are nullable and independently meaningful. Earlier versions of this scope considered using an organization-code component, but that adds implementation cost:

- new organization code field
- code assignment/backfill
- reserved sentinel for org-less jobs
- extra counter dimensions
- more decisions around which organization axis anchors the ID

The simpler global daily sequence avoids those costs while still fixing the original UUID exposure problem.

Tradeoff:

- `YYMMDDSSSSS` reveals the job creation date.
- The sequence is a bare monotonic integer that resets at UTC midnight. Anyone who sees two same-day operational IDs can calculate the exact number of jobs created between them. If someone observes the first and last IDs for a UTC day, they can infer the exact platform-wide job count for that day.
- It does not reveal raw UUIDs or internal database identity.

This volume visibility is an explicit product/business tradeoff accepted for the current scope in exchange for lower implementation cost.

## Backend Data Model

### Jobs

Add operational ID fields.

```text
jobs
- id UUID primary key
- operational_id string unique not null
- operational_year integer not null
- operational_month integer not null
- operational_day integer not null
- operational_sequence integer not null
```

Notes:

- Store `operational_id` as variable-width text/varchar, not fixed `char(11)`.
- Treat `SSSSS` as the minimum zero-padded sequence width, not a hard cap.
- If the platform exceeds 99,999 jobs in one day, the sequence may grow past five digits.
- Search, validation, and UI display should not hard-code 11 characters as an invariant.

### Job Counters

Add a counter table to generate sequence numbers safely.

```text
job_operational_id_counters
- year integer not null
- month integer not null
- day integer not null
- next_sequence integer not null
- created_at timestamp not null
- updated_at timestamp not null
```

Unique constraint:

```text
unique(year, month, day)
```

Scale note:

- The single `(year, month, day)` counter row means all job creation serializes through one row lock each UTC day.
- This is acceptable at current expected scale and keeps the implementation simple.
- If daily job volume makes this row a write bottleneck, revisit with counter sharding or a database sequence-based design.

## Database Constraints

Required:

```text
unique(jobs.operational_id)
```

Recommended additional constraint:

```text
unique(
  jobs.operational_year,
  jobs.operational_month,
  jobs.operational_day,
  jobs.operational_sequence
)
```

The constraints should make duplicate operational IDs impossible even under concurrent job creation.

## Generation Flow

On job creation:

1. Determine current year, month, and day in UTC, matching the codebase's existing timestamp convention.
2. In a database transaction, lock or upsert the matching counter row:

```text
year + month + day
```

3. Read `next_sequence`.
4. Increment `next_sequence`.
5. Render the operational ID:

```text
YY + MM + DD + sequence_padded
```

6. Save the job with the generated operational fields.
7. Commit the transaction.

Example:

```text
year = 2026
month = 7
day = 21
sequence = 1

YY = 26
MM = 07
DD = 21
sequence_padded = 00001

operational_id = 26072100001
```

## Stability Behavior

This codebase does not currently have a job-transfer-between-companies feature. Network dispatch is modeled through separate origin, customer-owner, and fulfillment axes.

If origin, fulfillment, or ownership fields change after creation:

- Keep `jobs.id` unchanged.
- Keep `jobs.operational_id` unchanged.
- Update only the appropriate tenancy/dispatch fields.

This makes the operational ID a stable public reference for the lifetime of the job.

## API Changes

### Create Job

Job creation responses should include:

```json
{
  "id": "internal-uuid",
  "operationalId": "26072100001"
}
```

Public/user-facing clients should display `operationalId` as the job reference.

### Get/List Jobs

Include `operationalId` in job DTOs.

Avoid labeling the internal UUID as "Job ID" in API responses consumed by the UI. The UUID can remain as `id` for internal routing and updates.

### Search

Support searching by operational ID.

Recommended search behavior:

- Exact match on `operational_id`.
- Optional partial match for admin/support tooling if existing search supports it.
- UUID lookup may remain available internally, but it should not be the normal user-facing search path.

## Migration Plan

For existing jobs:

1. Group jobs by creation year/month/day.
2. Sort each group by `created_at`, then UUID as a tie-breaker.
3. Assign sequences starting at `1`.
4. Generate operational IDs using the same format.
5. Populate operational fields.
6. Create or update counter rows so `next_sequence` is greater than the max assigned sequence for each day.

Important note:

- Do not use `origin_org_id`, `customer_owner_org_id`, or `fulfillment_org_id` as part of the operational ID.
- Do not invent a transfer migration. There is no current job-transfer feature to preserve.

## Existing Precedent and UI Leak

The codebase already separates public identifiers from internal IDs for customer tracking:

```text
jobs.tracking_token
```

That token is an opaque capability URL secret, while this operational ID is a readable reference, but the architectural category is similar: public identifier does not need to equal internal UUID.

The concrete user-facing leak this fixes is currently visible in:

```text
apps/technician-web/src/app/activity/page.tsx
```

That screen renders the raw `job.id` UUID under a "Job ID" label. After this work, technician-facing UI should render `operationalId` / `operational_id` for that label.

## Validation and Testing

Add tests for:

- New job receives an operational ID.
- Operational ID format is numeric and uses the configured minimum zero-padding.
- Daily sequence increments globally.
- Sequence resets naturally for a new day.
- Concurrent job creation cannot create duplicate IDs.
- Updating origin, fulfillment, or customer-owner fields does not change operational ID.
- Searching by operational ID returns the correct job.
- Migration assigns unique IDs to existing jobs.

## Acceptance Criteria

- New jobs have both an internal UUID and a separate operational ID.
- User-facing job references no longer expose UUIDs.
- Operational IDs follow `YYMMDDSSSSS`.
- Operational IDs are globally unique.
- Operational IDs are not anchored to any organization field.
- Operational IDs remain stable after origin, fulfillment, or ownership field changes.
- Duplicate IDs are prevented at the database level.
- Existing jobs are backfilled.
- Tests cover creation, concurrency, daily reset, field-change stability, search, and migration.

## Non-Goals

- No encryption or reversible cipher.
- No random/hash-based public code.
- No text prefix.
- No organization/company code.
- No hour component.
- No admin-managed prefix.
- No organization-managed reset.
- No manual operational ID editing.
- No change to UUID as the internal database identity.
